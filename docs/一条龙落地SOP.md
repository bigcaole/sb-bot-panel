# sb-bot-panel 一条龙落地 SOP（最少切换版）

适用目标：从零部署到可对外提供订阅，尽量减少管理端/节点端来回切换。

## 1. 总体原则

- 管理服务器先稳定：先完成 `controller + bot + HTTPS + 验收`。
- 节点服务器逐台接入：每台节点只做一次“快速配置”。
- 参数对齐靠管理端下发：优先用管理菜单 `14` 同步，不手改文件。
- 验证统一收口：最后只在管理服务器做状态确认。

## 2. 管理服务器（只做一次）

```bash
cd /root/sb-bot-panel
git pull --ff-only origin main
sb-admin
```

按顺序执行：

1. `23` 安装/重装（首次）
2. `22` 组件自检与自动修复
3. `12` 一键验收自检（建议带 API）

说明：

- 后续升级只用 `24`（更新，复用旧配置），不需要重填参数。
- `23` 是安装/重装；`24` 才是更新。

## 3. 新节点接入（最少切换，推荐流程）

### A. 先在管理服务器做准备

1. 用 Bot 或 API 创建节点（先建节点记录）。
2. `sb-admin` -> `14` -> `5`（新节点上线一条龙向导）
3. 抄下这 3 个值：
   - `controller_url`
   - `auth_token`（NODE 主 token）
   - `poll_interval`

### B. 切到节点服务器（每台只做一次）

```bash
cd /root/sb-bot-panel
git pull --ff-only origin main
bash scripts/install.sh --configure-quick
```

按提示填：

- `controller_url`：用上一步给出的值
- `node_code`：必须和管理端节点编码完全一致
- `auth_token`：用管理端给出的 NODE 主 token
- `tuic_domain`：需要 TUIC 才填
- `poll_interval`：默认 15 即可

### C. 回管理服务器收口

1. `sb-admin` -> `14` -> `1`（同步默认参数）
2. `sb-admin` -> `6`（查看状态 + 节点连接统计）
3. 确认新节点 `last_seen` 已上报

## 4. 用户开通（落地使用）

1. 创建用户
2. 给用户分配节点
3. 生成订阅链接并下发用户

如果浏览器打开订阅返回 `# no available links`：

- 通常是该用户还没分配任何启用节点。

## 5. 日常维护（高频）

管理服务器：

- 更新：`sb-admin` -> `24`
- 安全加固：`15`
- Token 操作：`16`
- 组件自检修复：`22`
- 一键验收：`12`

节点服务器：

- 更新代码并复用配置：`sb-node` -> `21`
- sing-box 缺失/损坏：`sb-node` -> `23`
- 证书检查：`sb-node` -> `9`
- 强制刷新证书：`sb-node` -> `10`

## 6. 证书与续期（节点端）

- 只要配置了 `tuic_domain`，`sing-box` 会使用内置 ACME 自动申请/续期。
- 项目里的 `sb-cert-check.timer` 是健康检查与告警辅助，不替代 sing-box 自身续期机制。
- 常用检查路径：
  - 菜单 `9` 看证书状态
  - 菜单 `8` 看 sing-box 日志
  - 菜单 `10` 做一次手动刷新

## 7. 常见报错快速处理

- `sing-box 未安装/服务不存在`
  - 节点菜单 `23` 安装/更新 sing-box
  - 再执行菜单 `1` 或 `21` 补齐服务

- `401 unauthorized`（节点拉取失败）
  - 管理菜单 `16 -> 3` 查看完整 token（有二次确认）
  - 确认节点 `auth_token` = `NODE_PRIMARY`
  - 管理菜单 `14 -> 2` 再同步一次节点 token

- `429 rate_limited`
  - 减少短时间连续点击
  - 检查管理端限流配置是否过低

- 删除节点失败（有进行中任务/绑定）
  - 先清理任务队列或用户绑定，再删节点

## 8. 不要做的事

- 不要把“更新”和“重装”混用。
- 不要手动改多处 token 后不做同步。
- 不要跳过管理端最后的收口验证（菜单 `6` + `12`）。

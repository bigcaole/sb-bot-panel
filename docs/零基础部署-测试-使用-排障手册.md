# sb-bot-panel 零基础部署-测试-使用-排障手册

适用对象：

- 没有开发经验、只会基础命令复制粘贴的管理员
- 一台管理服务器（运行 controller + bot）
- 一台或多台节点服务器（运行 sing-box + sb-agent）

本文目标：

1. 从 0 到 1 完成部署
2. 完成全链路测试（创建用户、绑定节点、拿订阅、节点在线监控）
3. 掌握日常操作和常见故障排查

---

## 1. 你要先理解的最小架构

本项目分成两类机器：

1. 管理服务器
   - 运行 `controller`（HTTP API + SQLite）
   - 运行 `bot`（Telegram 机器人）
2. 节点服务器
   - 运行 `sing-box`（真正承载代理入站）
   - 运行 `sb-agent`（从 controller 拉配置并热更新 sing-box）

数据流：

1. 你在 Telegram bot 上点按钮
2. bot 调用 controller API，写入数据库
3. 节点上的 agent 定时请求 `/nodes/{node_code}/sync`
4. agent 渲染 sing-box 配置并生效
5. 用户通过订阅链接拿到节点链接使用

## 2. 部署前准备清单

### 2.1 服务器准备

建议至少：

- 管理服务器：Debian 12，1C2G+
- 节点服务器：Debian 12，1C1G+

都需要：

- root 权限
- 可访问公网
- 时间同步正常（`timedatectl status`）

### 2.2 域名准备（用于 TUIC 证书）

如果你要启用 TUIC，请准备一个域名并保证：

1. A 记录指向节点公网 IP
2. Cloudflare 代理关闭（小黄云置灰）
3. 节点放行 UDP 端口（默认 24443，建议高位端口）

### 2.3 Telegram Bot 准备

你需要准备：

- `BOT_TOKEN`
- 管理员 `chat_id`（可通过 bot 的 `/whoami` 拿到）

## 4. 管理服务器部署（controller + bot）

### 4.1 启动中文菜单

```bash
cd /root/sb-bot-panel
sudo bash scripts/admin/menu_admin.sh
```

### 4.2 首次安装

菜单里输入 `24`（安装/重装）。

安装脚本会做：

1. 安装依赖（python3-venv、git、curl、jq 等）
2. 创建 venv 并安装 `requirements.txt`
3. 交互写入 `.env`
4. 写入 systemd：
   - `sb-controller.service`
   - `sb-bot.service`
5. 启动并设为开机自启

### 4.3 配置项怎么填（重要）

安装过程中重点字段：

- `CONTROLLER_PORT`：默认 `8080`
- `CONTROLLER_URL`：bot 调 controller 的地址，通常 `http://127.0.0.1:8080`
- `CONTROLLER_PUBLIC_URL`：可选，对外地址（给外部参考）
- `PANEL_BASE_URL`：建议填你的域名（bot 展示订阅链接时使用）
- `AUTH_TOKEN`：可选，保护 controller 接口（除 `/health`、`/sub/*`、文档页），留空等于关闭
- `BOT_TOKEN`：必填
- `ADMIN_CHAT_IDS`：建议填你的 chat_id（逗号分隔）
- 注意：`ADMIN_CHAT_IDS` 留空时，任何 Telegram 账号都能使用管理菜单
- `MIGRATE_DIR`：默认 `/var/backups/sb-migrate`

### 4.4 安装后立即检查

```bash
systemctl status sb-controller --no-pager
systemctl status sb-bot --no-pager
curl -sS http://127.0.0.1:8080/health
```

预期：

- 两个服务都是 `active (running)`
- `/health` 返回 `{"ok":true}`

## 5. 节点服务器部署（sing-box + agent）

### 5.1 启动节点菜单

```bash
cd /root/sb-bot-panel
sudo bash scripts/menu.sh
```

### 5.2 首次安装

菜单输入 `1`（配置；首次安装或后续更新都从这里进入）。
进入后可选：
- `1) 快速配置`
- `2) 高级变量设置向导`
- `3) 参数单项修改`（按 `中文说明｜参数名` 列表点选某一项直接改）

安装脚本会做：

1. 安装依赖和 sing-box
2. 创建 `/opt/sb-agent` 与 venv
3. 写入 `/etc/sb-agent/config.json`
4. 配置 UFW
5. 写入 systemd：
   - `sb-agent.service`
   - `sb-cert-check.service`
   - `sb-cert-check.timer`（可选启用）

### 5.3 节点配置项如何填写

- `controller_url`：管理服务器地址，例如 `http://管理服务器IP:8080`
- `node_code`：节点代码，例如 `N1`
- `auth_token`：优先使用管理服务器 `.env` 的 `NODE_AUTH_TOKEN`（若未拆分再用 `AUTH_TOKEN`）
- `tuic_domain`：TUIC 域名，不启用可留空
- `acme_email`：启用 TUIC 时必填
- `tuic_listen_port`：默认 `24443`（建议高位 UDP 端口）
- `poll_interval`：默认 `15`

### 5.4 节点安装后检查

```bash
systemctl status sb-agent --no-pager
systemctl status sing-box --no-pager
journalctl -u sb-agent -n 100 --no-pager
```

## 6. 首次业务初始化（Telegram 中操作）

### 6.1 进入 bot

给你的 bot 发送：

- `/start`
- `/whoami`（拿 chat_id）

### 6.2 创建节点

路径：`节点管理 -> 新增节点`

按向导输入：

1. `node_code`（例如 N1）
2. `region`（例如 JP）
3. `host`（节点 IP 或域名）
4. `reality_server_name`（可选）
5. `tuic_port_start` / `tuic_port_end`（端口池）

说明：

- 当前 TUIC 分享链接默认按 A 模式使用 `tuic_listen_port`（通常 24443）
- 端口池字段仍保留以兼容未来 B 模式

### 6.3 创建用户

路径：`用户管理 -> 创建用户`

按向导输入：

1. 备注名
2. TUIC 端口（当前保留字段）
3. 限速
4. 有效天数
5. 确认创建

### 6.4 绑定节点

路径：`用户管理 -> 节点分配`

流程：

1. 选用户
2. 分配节点
3. 确认

## 8. 节点在线监控使用说明（你关心的重点）

### 8.1 监控开关

在 bot 中进入：

`节点管理 -> 节点列表 -> 点某个节点 -> 开启节点监控`

每个节点可独立开关。

### 8.2 工作机制

1. agent 调用 `/nodes/{node_code}/sync` 时，controller 会更新 `last_seen_at`
2. bot 每隔 `BOT_NODE_MONITOR_INTERVAL` 秒轮询 `/nodes`
3. 若某节点 `monitor_enabled=1` 且超过阈值未心跳，则判定离线

### 8.3 告警规则

- 在线：不提醒
- 离线：只提醒一次
- 恢复：再提醒一次

### 8.4 建议配置（管理服务器 `.env`）

```env
ADMIN_CHAT_IDS=你的chat_id
BOT_NODE_MONITOR_INTERVAL=60
BOT_NODE_OFFLINE_THRESHOLD=120
```

重启：

```bash
systemctl restart sb-bot
```

### 8.5 监控功能验证步骤

1. 开启节点监控
2. 节点服务器执行：`systemctl stop sb-agent`
3. 等 1~2 分钟，应该收到“节点掉线”
4. 执行：`systemctl start sb-agent`
5. 再等 1 分钟，应该收到“节点恢复”

## 10. 常见故障与处理

### 10.1 bot 点按钮没反应

检查：

```bash
systemctl status sb-bot --no-pager
journalctl -u sb-bot -n 200 --no-pager
```

常见原因：

1. `BOT_TOKEN` 填错
2. 代码未更新（服务器还在旧版本）
3. 网络无法访问 Telegram

---

### 10.2 controller 启动失败

检查：

```bash
systemctl status sb-controller --no-pager
journalctl -u sb-controller -n 200 --no-pager
```

常见原因：

1. Python 依赖未安装完整
2. 代码版本不一致
3. `.env` 或端口冲突

---

### 10.3 节点长期显示离线

在节点服务器检查：

```bash
systemctl status sb-agent --no-pager
journalctl -u sb-agent -n 200 --no-pager
cat /etc/sb-agent/config.json
```

重点看：

1. `controller_url` 是否可达
2. `node_code` 是否和管理端一致
3. 若启用鉴权，`auth_token` 是否和管理端一致
4. 节点到管理服务器 8080 是否放通

---

### 10.4 订阅里没有 vless 链接

原因通常是节点缺少 REALITY 参数之一：

- `reality_server_name`
- `reality_public_key`
- `reality_short_id`

处理：

1. 在 bot 节点详情里配置 REALITY 参数
2. 再 curl 明文订阅确认

---

### 10.5 TUIC 连接失败

检查顺序：

1. 域名解析是否指向节点公网 IP
2. Cloudflare 小黄云是否关闭
3. 节点 `24443/udp`（或自定义端口）是否放行
4. `tuic_server_name` 是否设置合理
5. `journalctl -u sing-box -n 200 --no-pager`

---

### 10.6 备份/迁移接口 401

如果启用了 `AUTH_TOKEN`：

1. bot 的 `AUTH_TOKEN` 必须一致
2. 手动 curl 必须带 Bearer 头

示例：

```bash
curl -i -X POST http://127.0.0.1:8080/admin/backup \
  -H "Authorization: Bearer 你的AUTH_TOKEN"
```

---

### 10.7 Caddy 启动失败：`:443 bind: address already in use`

典型日志：

- `listen tcp :443: bind: address already in use`

这不是配置语法错误，而是 **443 端口被其他进程占用**。

排查：

```bash
ss -lntp | grep ':443'
systemctl status nginx apache2 caddy --no-pager
```

处理：

1. 若当前阶段不需要 HTTPS：将 `ENABLE_HTTPS=0`，重启 controller/bot，先完成功能迁移。
2. 若必须启用 HTTPS：
   - 停掉占用 443 的服务（如 nginx/apache）
   - 保留 caddy 独占 443
   - 再执行 `systemctl restart caddy`
3. 若仍失败，查看最近日志：

```bash
journalctl -u caddy -n 200 --no-pager
```

---

### 10.8 节点更换公网 IP（脱敏示例，按这个模板做）

适用场景：节点机器没变，只是公网 IP 变了。  
示例（脱敏）：

- `node_code=JP1`
- 旧 IP：`198.51.100.10`
- 新 IP：`203.0.113.25`

#### 步骤 A（管理服务器，必做）

1. 在 bot 进入：`节点管理 -> 节点列表 -> JP1`
2. 点击 `修改入口（影响两种协议）`，输入新 IP：`203.0.113.25`
3. 点击 `设置节点来源IP白名单`，输入新 IP：`203.0.113.25`
4. 管理菜单 `1 -> 4 参数单项修改`，更新：
   - `CONTROLLER_PORT_WHITELIST`：加入 `203.0.113.25/32`（旧 IP 可移除）
   - `SECURITY_BLOCK_PROTECTED_IPS`：加入 `203.0.113.25`（避免误封）

说明：

- 第 2 步会更新节点 `host`（订阅链接里的入口地址）
- 第 3 步会更新节点 `agent_ip`（节点访问收敛/来源识别）

#### 步骤 B（DNS，按需）

如果该节点启用了 TUIC 域名（如 `jp1.example.com`）：

1. 把该域名 A 记录指向新 IP `203.0.113.25`
2. Cloudflare 若在前面，确保关闭代理（灰云）

#### 步骤 C（节点服务器，通常只核对）

1. 核对 `/etc/sb-agent/config.json`：
   - `controller_url`：应是管理服务器地址（通常不用改）
   - `node_code`：必须还是原值（如 `JP1`）
   - `auth_token`：应与管理端当前生效 `NODE_AUTH_TOKEN` 一致
2. 重启服务：

```bash
systemctl restart sb-agent
systemctl restart sing-box
```

#### 步骤 D（验收，建议全做）

管理服务器：

```bash
set -a; . /root/sb-bot-panel/.env; set +a
curl -fsSL -H "Authorization: Bearer ${ADMIN_AUTH_TOKEN%%,*}" \
  "http://127.0.0.1:${CONTROLLER_PORT:-8080}/nodes/JP1" | jq '.host,.agent_ip,.last_seen_at'
```

节点服务器：

```bash
journalctl -u sb-agent -n 120 --no-pager | grep -Ei '401|unauthorized|sync' || true
systemctl status sing-box --no-pager -n 40
```

预期：

- 管理端 `host/agent_ip` 都是新 IP
- `last_seen_at` 持续刷新
- `sb-agent` 日志不再出现 401/unauthorized
- `sing-box` 为 `active (running)`

#### 常见失败点（本案例根因）

1. 只改了节点来源白名单，没改 `host`  
   - 现象：订阅里仍是旧入口地址
2. 新 IP 没加入 `CONTROLLER_PORT_WHITELIST`  
   - 现象：节点无法拉取 sync
3. 节点 `auth_token` 与管理端当前 `NODE_AUTH_TOKEN` 不一致  
   - 现象：`sb-agent` 循环报 401
4. TUIC 域名 DNS 还指向旧 IP  
   - 现象：证书/连接异常
